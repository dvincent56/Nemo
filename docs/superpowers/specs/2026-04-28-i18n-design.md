# Internationalisation de l'interface — Design

**Date :** 2026-04-28
**Statut :** Spec validée, prêt pour plan d'implémentation

## Contexte et motivation

La ROADMAP identifie l'i18n comme premier item de la Phase 4 et la marque comme prioritaire ("sinon ingérable plus tard"). Aujourd'hui, l'application web est intégralement codée en français en dur : ~120 fichiers TSX, métadonnées Next, manifest PWA, et catalogues de données métier (`game-balance.json`) contiennent tous des strings FR directement dans le code ou les JSON. Aucune infrastructure i18n n'est en place (pas de `next-intl`, pas de structure `[locale]/`, pas de fichiers de messages).

Bénéfices de cette livraison :
- Débloque la cible internationale (positionnement vs Virtual Regatta exige au minimum FR/EN/ES/DE).
- Pose une **règle dure** "plus jamais de string en dur côté UI" applicable dès le merge — chaque nouveau wording devra obligatoirement passer par une clé de traduction.
- Force la séparation propre entre données de balance (chiffres) et données de catalogue (textes affichés), ce qui assainit `game-balance.json`.
- Confirme et formalise la neutralité linguistique du backend (engine émet des codes, UI traduit).

## Périmètre

Cette livraison met en place :
- Infrastructure `next-intl` complète : middleware, routing préfixé, détection de locale, persistance.
- Extraction de **toutes** les strings UI vers des fichiers de messages structurés.
- 4 locales supportées : `fr`, `en`, `es`, `de`. **Seul `fr` est réellement traduit** dans cette vague ; `en`, `es`, `de` sont des stubs (copie identique du contenu FR) à traduire ultérieurement.
- Refacto du `game-balance.json` pour extraire `name`/`description` vers un nouveau fichier `boat-catalog.{locale}.json`.
- Audit et neutralisation linguistique du backend (API, WebSocket, engine notifications).
- Garde-fous automatiques : règle ESLint `react/jsx-no-literals` + script CI de complétude des traductions.
- Mise à jour de la page `/cookies` pour mentionner le cookie `NEMO_LOCALE`.
- Mise à jour du runbook `adding-a-boat-class.md` pour intégrer le catalogue.

Hors scope (à traiter séparément) :
- Traduction réelle vers en/es/de (vague ultérieure, dépend d'un budget/process traduction).
- Internationalisation des emails Cognito et transactionnels (sera traitée en Phase 4 quand Cognito sera câblé).
- Internationalisation de la future page `/admin/system` (Phase 5).
- Logs serveur (restent en anglais technique, non user-facing).
- Synchronisation du bloc `swell` entre les deux copies de `game-balance.json` (divergence pré-existante connue, hors scope explicite).

## Définitions

- **Locale** : code à 2 lettres ISO 639-1 (`fr`, `en`, `es`, `de`).
- **Locale source** : `fr`. C'est la seule réellement traduite dans cette vague et la base depuis laquelle les autres seront générées.
- **Stub locale** : fichier `messages/{locale}.json` dont le contenu est identique à `messages/fr.json` (copie verbatim). Permet l'extraction immédiate sans bloquer sur la traduction.
- **Catalogue** : `boat-catalog.{locale}.json`. Fichier qui mappe chaque boat class et chaque upgrade catalog ID à ses strings affichables (`name`, `description`, `tagline`).
- **Message** : entrée d'un fichier `messages/{locale}.json`, identifiée par une clé hiérarchique (ex. `marina.boat.upgradeAvailable`).
- **Code événement** : identifiant stable émis par le backend (ex. `wind-shift`, `zone-violation`, `mark-passed`) qui est résolu côté UI via la section `events.*` des messages.

## Architecture

### Library

`next-intl` v4.x. Compatible Next.js 16.2.3 et React 19.2. À vérifier en smoke test en tout début de PR 1 ; si incompatibilité, fallback vers `next-international` (alternative plus légère).

### Routing

Structure cible :
```
apps/web/src/app/
  [locale]/
    layout.tsx
    page.tsx                  # ex-/
    login/
    races/
    marina/
      [boatId]/
        customize/
      inventory/
    play/[raceId]/
    ranking/
      races/
      [raceId]/
    profile/
      [username]/
      settings/
      social/
    team/[slug]/
    news/[slug]/
    cgu/
    cookies/
    legal/
    privacy/
  api/                        # reste à la racine, hors locale
  not-found.tsx               # global
  layout.tsx                  # racine minimal (juste fournir <html><body>)
```

URLs : toutes préfixées (`/fr/marina`, `/en/marina`, `/es/marina`, `/de/marina`).

Middleware (`apps/web/src/middleware.ts`) :
- Intercepte les requêtes sans préfixe locale (`/marina`, `/`).
- Détermine la locale cible selon l'ordre :
  1. `user_settings.locale` si user authentifié (cookie de session Cognito présent).
  2. Cookie `NEMO_LOCALE` si présent.
  3. Header `Accept-Language` matché contre `['fr','en','es','de']` (best-fit).
  4. Fallback `fr`.
- Redirige en 301 (permanent) vers `/{locale}/...`.

### Détection et persistance

- **Cookie `NEMO_LOCALE`** : durée 1 an, `SameSite=Lax`, `Secure` en prod, `HttpOnly` désactivé (le client en a besoin pour le sélecteur côté React).
- **DB `user_settings.locale`** : nouvelle colonne `varchar(2) NOT NULL DEFAULT 'fr'`, à ajouter dans la migration des `user_settings` (déjà identifié comme gap dans le backlog backend).
- **Synchro** : à chaque changement explicite via le sélecteur de langue, on update :
  1. Le cookie `NEMO_LOCALE`.
  2. Si user authentifié : `PATCH /api/v1/users/me/settings` avec `{ locale: '...' }`.
  3. Le `useTransition` côté React pour relancer le rendu serveur dans la nouvelle locale.

### Sélecteur de langue UI

Présent dans :
- Le footer global (toujours visible, même non-authentifié).
- La page `/profile/settings` (section "Préférences").

Visualisation : 4 boutons `FR / EN / ES / DE` avec drapeau ou code, le bouton actif est highlight. Pas de menu déroulant (4 langues, ça tient).

## Structure des fichiers de traduction

### Emplacement

```
apps/web/messages/
  fr.json    # source de vérité
  en.json    # stub = copie de fr.json
  es.json    # stub = copie de fr.json
  de.json    # stub = copie de fr.json
```

### Namespacing

Calqué 1:1 sur la structure des routes pour minimiser la charge cognitive lors d'ajout de clés :

```jsonc
{
  "common": {
    "actions": { "save": "Enregistrer", "cancel": "Annuler", ... },
    "states":  { "loading": "Chargement…", "empty": "Aucun élément", ... },
    "units":   { "knots": "nds", "meters": "m", ... },
    "nav":     { "marina": "Marina", "races": "Courses", ... }
  },
  "home":    { ... },
  "login":   { ... },
  "races":   { "list": { ... }, "filters": { ... } },
  "marina":  { "list": { ... }, "boat": { ... }, "inventory": { ... }, "customize": { ... } },
  "play":    { "hud": { ... }, "compass": { ... }, "timeline": { ... }, "panels": { ... } },
  "ranking": { "season": { ... }, "races": { ... }, "race": { ... } },
  "profile": { "main": { ... }, "settings": { ... }, "social": { ... }, "public": { ... } },
  "team":    { ... },
  "news":    { "list": { ... }, "article": { ... } },
  "legal":   { "cgu": { ... }, "privacy": { ... }, "cookies": { ... }, "legal": { ... } },
  "errors":  { "network": "...", "auth": "...", "validation": { ... } },
  "events":  { "wind-shift": "...", "zone-violation": "...", "mark-passed": "..." },
  "meta":    { "title": "...", "description": "...", "manifestName": "..." }
}
```

### Conventions de nommage

- `kebab-case` pour les segments (`compass.target-cone`).
- Préfixes sémantiques internes : `actions.*`, `labels.*`, `messages.*`, `errors.*`.
- Profondeur maximale 3 niveaux.
- Section `events.*` réservée aux codes émis par le backend (matérialise le contrat backend → UI).

### ICU MessageFormat

Utilisation native de next-intl pour :
- Pluralisation : `"You have {count, plural, =0 {no boats} one {# boat} other {# boats}}"`.
- Interpolation : `t('marina.welcome', { name: user.username })` → `"Bienvenue, {name}"`.

### Typage TypeScript

`next-intl` génère les types à partir de `messages/fr.json` (locale source). Une clé inexistante = erreur de typecheck à la compile, pas un crash runtime.

### Catalogue séparé

Le fichier `boat-catalog.{locale}.json` n'est **pas** dans `messages/` car structuré différemment (mapping par ID, pas par namespace/clé). Voir section dédiée plus bas.

## Refactor du game-balance.json et catalogue

### État actuel

```json
// apps/web/public/data/game-balance.json
{
  "boats": {
    "class40": {
      "hulls": {
        "standard": {
          "name": "Coque Class40 Standard",
          "description": "Coque de série, polyvalente et fiable...",
          "speedBonus": 0, "drag": 1.0, ...
        }
      }
    }
  }
}
```

`name` et `description` sont en français en dur dans **deux** fichiers (web + engine) : ~100+ entrées (toutes les coques, mâts, voiles, préparations de toutes les classes).

### État cible

`game-balance.json` (web et engine, **les deux** versions) — pure balance, zéro texte :
```json
{
  "boats": {
    "class40": {
      "hulls": {
        "standard": {
          "speedBonus": 0, "drag": 1.0, ...
        }
      }
    }
  }
}
```

`apps/web/public/data/boat-catalog.{locale}.json` — textes uniquement :
```json
{
  "boats": {
    "class40": {
      "name": "Class40",
      "tagline": "Le monocoque polyvalent du large",
      "hulls": {
        "standard": {
          "name": "Coque Class40 Standard",
          "description": "Coque de série, polyvalente et fiable..."
        }
      },
      "masts": { ... },
      "sails": { ... },
      "preparations": { ... }
    },
    "imoca60": { ... }
  },
  "upgradeCategories": {
    "hull":  { "label": "Coque" },
    "mast":  { "label": "Mât" },
    "sails": { "label": "Voiles" },
    "prep":  { "label": "Préparation" }
  }
}
```

### Schéma Zod

Nouveau `BoatCatalogZ` dans `@nemo/game-balance` :
- Dérivé du tuple `BOAT_CLASSES` (alignement avec `BoatClassZ` existant).
- `Record<BoatClass, BoatCatalogEntryZ>` exhaustif → ajouter une classe sans cataloguer = erreur typecheck.
- Validé au chargement côté UI (`useBoatCatalog()`) et côté script `i18n:check`.

### Loader UI

Nouveau hook `useBoatCatalog()` :
- Charge `boat-catalog.{currentLocale}.json` via `fetch` + cache module-level.
- Tous les composants qui affichent un nom/description de boat ou upgrade migrent de `game-balance.json` vers `useBoatCatalog()`.

### Stubs en/es/de

À la création, `boat-catalog.{en,es,de}.json` = copie identique de `boat-catalog.fr.json`. Aucune traduction dans cette vague.

### Synchronisation des deux copies de game-balance

Le CLAUDE.md note la divergence pré-existante sur le bloc `swell`. Cette refacto **ne touche pas** à `swell`. Elle extrait uniquement `name`/`description` des deux fichiers, ce qui réduit mécaniquement la surface de divergence sans la résoudre.

### Runbook

`docs/runbooks/adding-a-boat-class.md` mis à jour pour ajouter une étape : créer l'entrée dans les 4 fichiers `boat-catalog.{locale}.json`. La copie fr→{en,es,de} reste manuelle dans ce runbook (cas trop rare pour automatiser).

## Neutralité linguistique du backend

### Principe

Aucun service backend (game-engine, weather-engine, ws-gateway, API Fastify) ne renvoie de string traduisible vers le client. Toujours un **code stable** que l'UI traduit via `events.*` ou `errors.*`.

### Audit préalable (à faire en début de PR 7)

Grep exhaustif sur `apps/game-engine/`, `apps/ws-gateway/`, `apps/web/src/app/api/` pour lister toutes les strings FR qui sortent côté client. Le projet est probablement déjà rigoureux mais ne pas le supposer.

### Surfaces et actions

| Surface | Vérification | Action si non conforme |
|---|---|---|
| API Fastify — réponses d'erreur | Body contient un code (ex. `BOAT_NOT_OWNED`), pas une phrase FR | Migrer vers `{ code: string, details?: object }` |
| API Fastify — validations Zod | Les `message: "..."` ne contiennent pas de FR | Migrer vers code |
| WS events (ws-gateway → client) | Payloads en codes (déjà attendu) | Audit + correction si trouvé |
| game-engine notifications | Notifs joueur émettent un type + params, pas une phrase | Refactor le cas échéant |
| Logs serveur | Restent en anglais technique | Pas de changement |
| Emails Cognito | Hors scope, à traiter en Phase 4 | — |

### Helper UI

Nouveau helper `tEvent(code, params?)` :
- Résout `events.{code}` dans les messages courants.
- Si la clé est absente : log warn + affichage du code brut (pas de crash).
- Tous les consumers de WS events / API errors passent par ce helper.

## Enforcement

### ESLint — `react/jsx-no-literals`

Configuration dans `apps/web/eslint.config.mjs` :
```js
{
  files: ['src/app/[locale]/**/*.{tsx,ts}', 'src/components/**/*.{tsx,ts}'],
  rules: {
    'react/jsx-no-literals': ['error', {
      noStrings: true,
      ignoreProps: false,
      noAttributeStrings: true,
      allowedStrings: ['—', '·', '•', '/', ':', '|', '×', '→', '↑', '↓', '+', '-'],
    }]
  }
}
```

Couvre :
- Texte JSX brut.
- Attributs textuels (`placeholder`, `alt`, `aria-label`, `title`).

Activation **progressive** : la règle ne s'active que sur les fichiers déjà migrés (sous `[locale]/`). Les fichiers en attente ne bloquent pas le build. Une fois la migration terminée (PR 7), le glob couvre l'ensemble.

### Script CI — `scripts/check-translations.ts`

Lancé en CI (job dédié) et localement via `pnpm i18n:check`.

Algorithme :
1. Parse récursivement `apps/web/src/**/*.{tsx,ts}` pour extraire :
   - `useTranslations('namespace')` puis tous les `t('key')` correspondants.
   - `getTranslations({ locale, namespace })` (server components).
   - `tEvent('code')` (helper events).
2. Construit `usedKeys`.
3. Charge `messages/{fr,en,es,de}.json`.
4. Pour chaque locale :
   - **Missing** : clé utilisée mais absente → **fail**.
   - **Orphan** : clé présente mais jamais utilisée → **warn** (pas fail, pour clés à venir).
   - **Inconsistent** : clé présente dans `fr.json` mais absente d'une autre locale → **fail** (la complétude est obligatoire, même en stub).
5. Rapport : `✓ 247 clés utilisées, 4 locales complètes` ou `✗ marina.newButton manquante dans en.json, es.json`.

### Convention sur les clés dynamiques

**Règle** : toujours utiliser des clés statiques (`t('marina.boat.title')`, jamais `t(\`marina.boat.${id}\`)`).

**Exception** : si un cas vraiment dynamique apparaît, documenter dans un fichier `apps/web/scripts/expected-dynamic-keys.txt` que le script `i18n:check` lit pour ne pas signaler ces clés comme orphelines.

### Pas de hook pre-commit

Le check tourne en CI uniquement. Pas de pre-commit pour ne pas ralentir chaque commit.

### Ordre des PRs vs ordre des locales

Toute la migration livre du français pur. Les fichiers `en.json`, `es.json`, `de.json` restent identiques au `fr.json` à chaque PR. Une 8e vague ultérieure (hors scope) commandera les vraies traductions.

### Hypothèse de durée

PR 1-2 : ~1 semaine chacune. PR 3-7 : 2-3 jours chacune en série. Total brut ~3-4 semaines de focus dédié, mergeable progressivement (pas de blocage des autres travaux).

## Risques et mitigations

| Risque | Mitigation |
|---|---|
| Régression visuelle sur `/play` (composant dense, animé) | PR 6 isolée, tests manuels ciblés (course en cours + replay) avant merge |
| Casse SEO temporaire sur les redirects | Redirects 301 (permanent), Google les absorbe en quelques jours, sitemap regénéré dans PR 2 |
| Conflits de merge avec features en cours (mockups, démock) | Coordination : PR 2 (déplacement physique) le plus tôt possible, puis chaque PR touche moins de fichiers |
| Incompatibilité next-intl + Next 16 | Smoke test au tout début de PR 1, fallback `next-international` si bloquant |
| Oubli de migration sur une route après merge de PR 2 | ESLint configuré sur `[locale]/**` empêche tout commit avec strings en dur sur les routes migrées ; les routes oubliées seraient identifiées car non déplacées |
| Clé manquante dans une locale stub | Script `i18n:check` fail le build CI, donc impossible de merger un PR avec locale incomplète |
| Sélecteur de langue casse l'état en cours (course live) | Le changement de locale via `useTransition` re-render serveur en gardant l'état client (pas de F5), à valider en PR 1 sur smoke test |

## Décisions différées (hors scope explicite)

- **Traduction réelle des locales en/es/de** : fait dans une vague ultérieure. Choix du process (DeepL automatique + relecture, ou traducteur humain) à trancher à ce moment.
- **i18n des emails** : à intégrer quand Cognito sera câblé (Phase 4).
- **i18n de la page `/admin/system`** : Phase 5.
- **Convention pour les contenus rich text traduits** (news, légales) : à voir si on garde des fichiers MDX par locale ou si on intègre dans les messages JSON. Décision reportée à la PR 4 (qui touchera ces pages).
- **Fallback gracieux si une clé manque en runtime** : en théorie le check CI empêche ça, mais on peut ajouter un fallback "afficher la clé brute" + Sentry warn si ça arrive en prod. À implémenter dans PR 1.
